# 扩展内核契约

定义 Garlic Claw extension kernel 的稳定边界。后续按此维护。

核心原则：内核保持简单，只留稳定边界和少量原语。SDK 提供作者体验层，但不把语法糖固化进 core。

## 1. 统一层

统一发生在 runtime contract，不在 authoring format。

- `plugin` — 宿主原生扩展协议
  - `packages/shared/src/types/plugin.ts`
  - `packages/server/src/plugin/plugin.module.ts`
  - `packages/plugin-sdk/src/authoring/`、`client/`、`host/`
- `MCP` — 外部能力/资源协议，通过 adapter 接到 kernel
  - `packages/server/src/execution/mcp/mcp.service.ts`
  - `packages/server/src/execution/tool/tool-registry.service.ts`
- `skill` — 兼容现有 skill 生态
  - `packages/server/src/execution/skill/skill-tool.service.ts`
  - `packages/server/src/execution/skill/skill-registry.service.ts`

作者写什么格式由各自生态决定，运行时只看最终暴露出来的 capability surface。

## 2. 内核原语

kernel 只保留两类一等原语：

- **action call** — plugin tools、MCP tools、native `skill` tool、host api methods
- **event subscription** — plugin hooks、chat/message/automation/tool/response/lifecycle hook 链

其他是边缘入口，不是第三套核心原语：
- `route` — HTTP ingress
- `cron` — host 调度入口
- `subagent` — 受控 workflow delegation，仍通过 host action 发起

## 3. Exported Surface

默认私有，只有显式导出的 surface 才能被发现或调用。

- plugin 只在 manifest 中声明了 `tools/hooks/routes/crons/config` 才进入 runtime
- MCP 只在配置启用并成功接入后，server/tool 才进入统一工具层
- skill 只在模型按需调用 `skill` 工具后，内容才送入上下文

扩展之间不允许裸对象直连。内核内部可以做 fast-path（builtin transport 进程内调用、lazy resolver、本地 snapshot 复用），但这些只是实现优化，不构成作者可依赖的语义。

## 4. Skill 的双面

skill 不再作为第三套厚 runtime：

- **对 AI**：prompt、policy、catalog discovery、lazy load，主要落在 `SkillToolService`
- **对 runtime**：load policy、asset inventory、native skill tool surface，主要落在 `SkillRegistryService` + `ToolRegistryService`。skill 触发的执行最终回到通用工具链

skill 依赖解析走 capability contract，不绑死某个扩展名。如果后续需要为特定生态补覆盖层，做成 override，不写成默认绑定。

## 5. 最小状态原语

kernel 只保留通用状态原语，不为每种能力新增专用核心面：

| 类别 | 原语 | 作用域 |
|------|------|--------|
| 私有 KV | `storage.*` | plugin / conversation / user |
| 临时状态 | `state.*` | plugin / conversation / user |
| 导出快照 | `provider.*`、`persona.*`、`kb.*`、`conversation.get`、`user.get`、`plugin.self.get` | — |
| 只追加日志 | `log.list`、`log.write`、`PluginPersistenceService` 事件方法 | — |

原则：优先复用统一原语，不默认为扩展新增专用 Prisma schema。只有宿主业务本身需要的一等实体才留在 core schema。

## 6. Builtin

builtin 是参考实现，不是特权实现。builtin 和 remote 共用同一套 plugin manifest/permission/hook/route/host api/runtime 语义。

已从 core 迁为 builtin consumer 的能力：memory、kb context、conversation title、automation、persona router、subagent delegate、governance/response/tool/lifecycle recorder 等。

后续原则：新能力先判断能否做成 plugin/MCP/skill/tool。builtin 优先作为"怎么消费内核"的样例，不成为堆 core 特判的借口。

## 7. 维护规则

新增能力的判断顺序：

1. 能不能只是一个新的 plugin / MCP / skill adapter？
2. 能不能只给现有 action/event 加 surface，不造新原语？
3. 能不能只用 `storage/state/exported snapshot/event log` 表达状态？
4. 前三步都不成立，才考虑进 core。

禁止：
- 造第三套扩展 runtime
- skill 反向绑死某插件格式
- MCP 绕开统一工具层直连聊天主链
- 为单个 builtin 新增 core 私有入口
- 把 fast-path 暴露成作者语义
