# 后端模型调用

本文档说明后端怎么配 provider、调模型、以及聊天/图片/停止/重试接口。

## 1. Provider 配置

### 1.1 配置文件

AI provider 配置放在 `config/ai/`。设了 `GARLIC_CLAW_AI_SETTINGS_PATH` 的话会读那个路径。

模板文件：

```
config/ai/providers/*.example.json
config/ai/host-model-routing.example.json
config/ai/vision-fallback.example.json
```

把模板复制出来改：

```bash
cp config/ai/providers/openai.example.json config/ai/providers/openai.json
```

### 1.2 Provider 类型

每个 provider 由一个 JSON 文件描述，核心字段是 `driver`，用它决定走哪个协议适配器。当前支持的 `driver`：

- `openai` → `@ai-sdk/openai`
- `anthropic` → `@ai-sdk/anthropic`
- `gemini` → `@ai-sdk/google`

其中 `openai` 固定走 `/v1/chat/completions`，不会落到 Responses API。

provider 的 `apiKey`、`baseUrl`、默认模型、模型列表都在各自的 JSON 文件里配。

## 2. 聊天接口

```
POST /api/chat/conversations/:id/messages
```

这个接口负责读 provider/model、处理图片、创建消息、以 SSE 流式返回生成结果。

### 2.1 请求体

```json
{
  "content": "分析这段 SQL",
  "parts": [{ "type": "text", "text": "分析这段 SQL" }],
  "provider": "openai",
  "model": "gpt-4o"
}
```

`content` 是纯文本摘要，`parts` 是结构化内容（支持 `text` 和 `image`）。不传 `provider/model` 的话会用上次对话的模型或默认模型。

## 3. SSE 事件

- `message-start` — 创建好的用户消息和 assistant 消息
- `status` — `pending / streaming / completed / stopped / error`
- `text-delta` — 文本增量
- `tool-call` / `tool-result` — 工具调用与结果
- `finish` / `error` — 结束或失败

## 4. 停止、重试、编辑、删除

- `POST /api/chat/conversations/:id/messages/:messageId/stop` — 停止生成中消息
- `POST /api/chat/conversations/:id/messages/:messageId/retry` — 重试最后一条 assistant 回复（可选 `{ provider, model }`）
- `PATCH /api/chat/conversations/:id/messages/:messageId` — 修改消息内容
- `DELETE /api/chat/conversations/:id/messages/:messageId` — 删除消息

## 5. 图片与 Vision Fallback

图片 part 格式：

```json
{ "type": "image", "image": "data:image/png;base64,...", "mimeType": "image/png" }
```

如果目标模型支持图片输入，直接送原图。不支持的话：
1. 查当前会话有没有同图缓存
2. 没缓存就走 `visionFallback` 配置的视觉模型转述
3. 把转述文本替换图片 part 送给聊天模型

转述失败会通过 SSE `error` 事件返回，不吞错。

## 7. 后端内部调模型

后端业务代码不要直接调某家 SDK。走统一调用层：

### 三层分工

| 场景 | 用这个 |
|------|--------|
| 完整聊天（落库、SSE、停止/重试） | `ConversationMessageLifecycleService` + `ConversationTaskService` |
| 一次性文本生成（分析、摘要） | `AiModelExecutionService` |
| 图片转述 | `AiVisionService` |

### AiModelExecutionService 用法

```ts
const result = await this.aiModelExecutionService.generateText({
  providerId: 'openai',
  modelId: 'gpt-4o',
  system: '你是 SQL 性能分析助手',
  messages: [{ role: 'user', content: [{ type: 'text', text: '分析这段 SQL' }] }],
  maxOutputTokens: 1024,
});
// result.text
```

参数覆盖顺序：`model.options` → `model.variants[variant]` → 调用时传入。

### generateText vs streamText

- `generateText` — 一次性生成，适合后台分析、测试连接、图片转述
- `streamText` — 流式生成，适合业务服务自己边生成边处理

### 图片缓存

`AiVisionService.resolveMessageParts()` 需要 `conversationId` 做同图缓存。非聊天场景传稳定逻辑 ID（如 `task:<taskId>`），别每次传随机 ID。

## 8. 新增 Provider

如果要加新的 provider 协议适配：

- catalog 目录：`packages/server/src/ai-management/ai-provider-catalog.ts`
- 运行时注册：`packages/server/src/ai-management/ai-management.service.ts`
- 协议族：`@ai-sdk/openai` / `@ai-sdk/anthropic` / `@ai-sdk/google`

目前只支持这三种协议族。业务代码不直接依赖 provider SDK。
